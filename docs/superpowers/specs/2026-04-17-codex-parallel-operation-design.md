# Codex 웹대시보드 병행 운영 설계

작성일: 2026-04-17

## 목적

Claude 웹대시보드와 Codex 웹대시보드를 같은 머신에서 동시에 운영하되, 포트 충돌뿐 아니라 데이터 저장소와 서비스 식별자까지 분리해 운영 실수 여지를 없앤다.

이번 설계의 목표는 다음 네 가지다.

- Codex 대시보드의 기본 포트를 `8617`로 고정한다.
- Claude 대시보드와 DB 파일, 백업 경로, 서비스 이름이 겹치지 않게 분리한다.
- `codex-web-dashboard` 라는 독립 실행 단위로 서비스 관리가 가능해야 한다.
- 같은 네트워크의 다른 기기/브라우저에서 `8617`로 접근 가능하고, 로그인 강제 설정이 실제로 적용되는지 확인할 수 있어야 한다.

## 운영 원칙

Codex 대시보드는 Claude 대시보드의 설정값 일부만 바꾼 파생 실행본이 아니라, 운영 식별자가 분리된 별도 인스턴스로 취급한다.

분리 대상은 다음과 같다.

- 포트: Codex 기본 포트는 `8617`
- DB 경로: Codex 전용 DB 파일 사용
- 백업 경로: Codex 전용 백업 디렉터리 사용
- 서비스 이름: `codex-web-dashboard`
- 실행 기준 문구: Codex 기준 브랜딩과 운영 문구 사용

이 원칙을 지키면 Claude 인스턴스가 `8765`에서 계속 동작하더라도 Codex 인스턴스를 별도로 시작, 중지, 재시작, 점검할 수 있다.

## 구성 요소

### 1. 포트 기본값

Codex 코드베이스의 기본 포트는 `8617`로 변경한다. 실행 시 `PORT`를 지정하지 않아도 Codex 인스턴스는 `8617`에서 올라와야 한다.

이 변경은 Codex 쪽 기본 동작을 고정하기 위한 것이며, Claude 대시보드의 기존 기본 포트나 실행 방식에는 영향을 주지 않는다.

### 2. DB 및 백업 경로 분리

Codex 인스턴스는 Claude 대시보드와 다른 DB 파일을 사용해야 한다. 경로명은 운영에서 봤을 때 Codex 전용임이 즉시 드러나야 한다.

예시 기준:

- DB 파일: `~/.codex/dashboard.db`
- 백업 경로: `~/.codex/dashboard-backups/`

핵심은 정확한 문자열보다도 Claude 쪽 기본 경로와 절대 충돌하지 않는 것이다. 기존 `.claude` 기준 경로를 그대로 재사용하면 병행 운영에서 데이터 혼선이 생기므로 허용하지 않는다.

### 3. 서비스 이름 분리

systemd 또는 유사한 서비스 관리 기준에서 Codex 인스턴스는 `codex-web-dashboard` 라는 이름으로 식별한다.

이 서비스 단위는 다음 요구를 만족해야 한다.

- Claude 서비스와 독립적으로 start/stop/restart 가능
- 로그 확인 시 Codex 서비스임이 즉시 식별 가능
- 포트 `8617`과 Codex 전용 경로를 사용하는 실행 정의 포함

### 4. 사용자 접근 확인 범위

접근 확인은 로컬 `localhost` 브라우저 수준에서 끝내지 않는다. 같은 네트워크의 다른 기기에서 접속 가능한지까지 포함한다.

확인 항목은 다음과 같다.

- 프로세스가 `0.0.0.0:8617` 또는 외부 접근 가능한 바인딩으로 기동
- 서버 머신의 방화벽 또는 네트워크 정책이 `8617` 접근을 막지 않음
- 다른 기기 브라우저에서 `http://<서버IP>:8617` 접속 가능
- 로그인 강제가 설정된 경우 비로그인 사용자는 보호된 API 또는 메인 화면에 직접 접근 불가

## 인증 확인 기준

로그인 강제 로직 자체가 코드에 존재하는지보다, 실행 중인 Codex 인스턴스에 실제 환경변수가 주입되어 있는지가 더 중요하다.

따라서 운영 확인은 다음 순서로 본다.

1. Codex 인스턴스 프로세스가 `8617`에서 실행 중인지 확인
2. 실행 프로세스 환경에 `DASHBOARD_PASSWORD`가 실제로 설정돼 있는지 확인
3. `/api/auth/me` 에서 `auth_required` 값 확인
4. 쿠키 없이 보호 API 접근 시 차단되는지 확인

이 검증을 통과해야만 “로그인 강제가 적용되어 있다”고 판단한다.

## 구현 범위

이번 변경의 구현 범위는 다음으로 한정한다.

- Codex 코드베이스 기본 포트 `8617` 반영
- Codex 전용 DB/백업 경로 분리
- `codex-web-dashboard` 서비스 식별자 기준 정리
- 병행 운영 및 사용자 접근 확인 절차 문서화

이번 범위에서 바로 하지 않는 항목은 다음과 같다.

- reverse proxy 또는 공인 도메인 구성
- TLS 인증서 발급/배치
- 외부 인터넷 공개 운영

## 성공 기준

다음 조건을 모두 만족하면 성공으로 본다.

- Claude 대시보드는 기존 포트에서 계속 사용 가능
- Codex 대시보드는 기본값만으로 `8617`에서 기동
- 두 인스턴스가 서로 다른 DB와 백업 경로를 사용
- 서비스 관리 단위가 `codex-web-dashboard`로 분리
- 같은 네트워크의 다른 기기에서 Codex 대시보드 접근 가능
- 로그인 강제 설정 시 비인증 접근 차단이 확인됨

## 위험 요소

### 1. 실행 중인 인스턴스 혼동

코드 저장소는 Codex용인데 실제 떠 있는 프로세스가 Claude 경로에서 실행 중일 수 있다. 이 경우 포트와 인증 확인 결과가 현재 저장소 변경과 무관해질 수 있다.

대응: 프로세스 경로, 포트, 환경변수를 항상 함께 확인한다.

### 2. 경로 분리 누락

포트만 바뀌고 DB/백업 경로가 기존 Claude 경로를 계속 쓰면 데이터 충돌이 발생한다.

대응: 기본 경로 자체를 Codex 전용으로 고정하고, 관리자 상태 응답이나 운영 문서에서 실제 경로를 확인 가능하게 한다.

### 3. 인증 설정 오인

로그인 화면 코드가 있어도 실제 실행 환경에 `DASHBOARD_PASSWORD`가 없으면 인증은 비활성화된다.

대응: 접근 확인 시 반드시 프로세스 환경과 `/api/auth/me` 응답을 함께 본다.
