# Codex Dashboard UX Redesign Design

## 목적

Codex Dashboard를 "기능 나열형 다기능 화면"에서 "운영 판단 중심 콘솔"로 재구성한다.

현재 UI는 기능 수와 정보량은 충분하지만, 첫 화면에서 사용자가 바로 판단해야 할 운영 상태와 비용 흐름이 동일한 톤으로 섞여 보인다. 이번 리디자인의 목표는 다음 두 가지다.

- 첫 화면 10초 안에 운영 상태를 파악할 수 있게 만든다.
- 탐색/분석/관리 기능은 유지하되 정보 구조를 더 명확하게 분리한다.

이번 작업은 기능 축소가 아니라 UI/UX 재배치다. 기존 API와 핵심 기능은 유지한다.

## 결정 사항

### 1. 제품 기본 진입 목적

- 기본 홈은 `운영 대시보드 중심`으로 재정의한다.
- 첫 진입 화면은 검색 도구가 아니라 운영 요약 화면이어야 한다.

### 2. 홈 상단의 성격

- 홈 최상단은 `운영 요약`형으로 설계한다.
- 상태와 비용 중 하나만 강조하지 않고, 둘을 동시에 읽게 만든다.

### 3. 내비게이션 구조

1차 내비게이션은 4개만 노출한다.

- `개요`
- `탐색`
- `분석`
- `관리`

세부 화면은 각 군집 내부에서 다시 노출한다.

### 4. 검색의 위치

검색은 두 위치를 모두 유지한다.

- `탐색` 탭의 기본 화면
- 어디서든 호출 가능한 전역 검색/명령 팔레트

즉, 검색은 독립 목적 화면이면서도 전역 도구다.

### 5. 시각 리디자인 범위

- 구조 재배치만 하지 않는다.
- 운영 콘솔에 맞게 색 체계, KPI 타이포, 카드 위계, 경고/상태 강조까지 함께 재설계한다.

## 정보 구조

### 1차 구조

#### 개요

운영 판단용 홈.

- 오늘 비용, 예산, 활성 세션, ingest/노드 상태
- 경고/이상징후
- 활성 프로젝트와 최근 흐름
- 분석 화면으로 이동하는 진입 블록

#### 탐색

검색과 세션 복기 작업 영역.

- 검색
- 세션 목록
- 대화/문맥 복기

#### 분석

시계열/분류별 분석 화면.

- 비용
- 모델
- 프로젝트
- 타임라인
- Subagent

#### 관리

운영 유지보수와 시스템 작업.

- Export
- Admin 상태
- 백업/보존
- 노드 관리
- 감사 로그

## 개요 홈 설계

`개요`는 운영 상태 판단을 끝내는 홈이어야 한다. 화면 리듬은 `상단 요약 → 위험 신호 → 현재 흐름 → 하위 분석 진입` 순서를 따른다.

### 상단: 핵심 운영 요약

최상단 KPI 블록은 다음을 포함한다.

- 오늘 비용
- 예산 진행률
- 활성 세션 수
- 원격 노드/수집 상태

이 영역은 숫자 위계가 가장 강해야 한다. 설명 텍스트보다 수치가 먼저 보이게 만든다.

### 중상단: 이상징후 / 주의 신호

운영자가 즉시 대응해야 하는 항목을 카드 또는 스트립으로 배치한다.

- 연결 끊김
- ingest 지연
- DB 팽창/free page 과다
- 예산 초과 위험

정상 상태와 경고 상태는 같은 무게로 보이면 안 된다. 경고는 더 높은 대비와 명확한 상태 색으로 분리한다.

### 중단: 지금 보고 있어야 할 흐름

현재 활동을 읽는 영역이다.

- 활성 프로젝트 TOP
- 최근 세션/agent-run 흐름
- 오늘 대비 증감

이 영역은 운영자가 "지금 무슨 일이 일어나는가"를 읽는 데 집중한다.

### 하단: 분석 진입 블록

세부 분석으로 내려가는 landing zone 역할을 한다.

- 비용 분석
- 타임라인
- 프로젝트 분석
- Subagent 분석

기존처럼 단순 메뉴 이동이 아니라, 각 영역의 현재 상태를 요약해서 클릭 동기를 만든다.

## 시각 시스템 재정의

### 기본 원칙

- 현재 Supanova 톤은 유지하되, 운영 콘솔에 맞게 더 기능적이고 상태 중심적으로 다듬는다.
- 모든 카드가 같은 존재감으로 보이지 않게 한다.
- "예쁜 대시보드"보다 "판단이 빠른 대시보드"를 우선한다.

### 색 체계

- 브랜드 액센트는 기존 emerald 축을 유지한다.
- 상태 색은 별도 체계로 분리한다.
  - 정상: emerald/cyan 범주
  - 주의: amber 범주
  - 위험: red 범주
  - 원격/분산 상태: cyan 또는 blue 범주
  - agent/subagent 계열: 보조 accent로 분리 가능
- 브랜드 액센트와 상태 색을 혼용하지 않는다.

### 타이포그래피

- KPI 숫자는 현재보다 더 크게, 더 무겁게, 더 명확한 자릿수 정렬을 갖는다.
- 숫자형 메트릭은 tabular 숫자 기준으로 읽히게 유지한다.
- 보조 라벨과 설명은 한 단계 더 줄여 메트릭 대비를 극대화한다.

### 표면과 카드

- 홈 화면 카드들은 3단계 위계를 갖는다.
  - 1급: 핵심 KPI
  - 2급: 경고/흐름
  - 3급: 진입 카드
- 같은 glass/bezel 문법은 유지하되, 위계별 padding, 대비, border intensity를 다르게 준다.

### 밀도 전략

- `개요`는 더 editorial하고 여백이 있는 구조
- `탐색`은 검색과 문맥 복기에 최적화된 작업 밀도
- `분석`은 dense하고 전문가용
- `관리`는 utilitarian하고 절제된 톤

즉, 뷰별로 다른 문법을 의도적으로 사용한다.

## UX 동작 원칙

### 전역 검색

- 전역 검색/명령 팔레트는 어디서든 즉시 열 수 있어야 한다.
- `탐색`에 진입하면 같은 검색이 더 넓은 작업 컨텍스트 안에서 열린다.
- 전역 검색은 빠른 점프용, 탐색 탭 검색은 조사/복기용으로 역할을 분리한다.

### 내비게이션

- 상단 1차 nav는 4개만 유지한다.
- 세부 뷰 전환은 탭, 서브필터, 또는 2차 segmented control로 처리한다.
- 현재의 과밀한 pill nav는 제거 대상이다.

### 홈의 역할 제한

- 홈은 상세 조사를 끝내는 곳이 아니다.
- 홈은 "문제가 있는가 / 어디를 봐야 하는가"를 결정하는 화면이다.

## 범위

### 포함

- 상단 내비게이션 재구성
- `개요` 홈 완전 재설계
- `탐색/분석/관리` 군집 구조 반영
- 검색의 전역 도구 + 탐색 기본 화면 역할 정리
- KPI/경고/진입 카드의 시각 위계 재설계

### 제외

- 백엔드 API 계약 변경
- 기능 삭제
- 데이터 모델 변경
- 현재 분석 기능 자체의 축소

## 검증 기준

리디자인 완료 후 다음을 만족해야 한다.

1. 첫 진입 시 상단 1차 nav는 `개요 / 탐색 / 분석 / 관리` 4개만 보여야 한다.
2. 홈 첫 화면에서 상태와 비용을 함께 읽는 운영 요약이 가장 먼저 보여야 한다.
3. 검색은 전역 호출과 탐색 탭 내부에서 모두 접근 가능해야 한다.
4. 경고/이상징후 블록은 일반 정보 카드보다 더 강한 시각 우선순위를 가져야 한다.
5. 기존 분석/관리 기능은 제거 없이 새 정보 구조 안에서 계속 접근 가능해야 한다.

## 구현 우선순위

1. 상단 nav 압축
2. `개요` 홈 재설계
3. 전역 검색과 탐색 화면 역할 정리
4. `분석` 군집 하위 IA 정리
5. 상태 색/KPI 위계/카드 톤 최적화

## 추천 구현 슬라이스

첫 구현 슬라이스는 `개요 + nav`만 다룬다.

이 슬라이스가 성공하면 사용자는 첫 진입에서 이미 제품의 중심을 다르게 인식하게 된다. 이후 `탐색`, `분석`, `관리`는 동일한 구조 원칙으로 순차 정리한다.
